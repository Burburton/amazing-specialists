# 系统状态机与对象模型设计

## 1. 文档目的

本文档用于定义“全自动产品研发闭环 MVP”中的**系统状态机**与**核心对象模型**，为后续的调度实现、任务执行、验证回环、验收汇总提供统一的数据结构与状态流转基础。

本文档解决的问题包括：

- 系统在每个阶段处于什么状态
- 状态之间如何迁移
- 什么事件会触发迁移
- 迁移时需要生成哪些对象
- 对象之间如何关联
- 对象落盘时采用什么目录结构
- 哪些字段是必须的，哪些字段可选
- 哪些状态允许自动推进，哪些必须等待人工验收
- 如何处理中断、失败、返工与重规划

---

## 2. 设计目标

本设计的目标不是“描述性文档”，而是作为系统实现的**状态与数据契约**，满足以下要求：

1. **可执行**
   - 能直接映射到代码里的枚举、类、数据库表或 YAML / JSON 文件

2. **可追踪**
   - 每个状态转换都可记录来源、原因、时间与责任方

3. **可回放**
   - 可以根据历史状态与事件恢复项目推进轨迹

4. **可扩展**
   - MVP 先支持单项目、单需求主线，后续可扩展到多项目、多里程碑并行

5. **可自动化**
   - 支持自动推进、失败返工、验收等待、异常升级

## 4. 状态机设计总原则

### 4.1 单向主流程，局部允许回退

系统整体主流程默认单向推进：

`接收需求 -> 澄清 -> 规格化 -> 规划 -> 执行 -> 验证 -> 验收`

但局部允许回退，例如：

- spec 不通过，回到 clarifying
- milestone 验收失败，回到 planning 或 execution
- task 测试失败，回到 in_progress

### 4.2 状态只表示“事实”，不表示“建议”

例如：

- `FAILED_TEST` 表示测试失败，是事实
- “建议返工”不应作为状态，而应写入 `recommended_action`

### 4.3 事件驱动，而不是函数隐式跳转

状态迁移必须由事件触发，例如：

- `requirement_clarified`
- `spec_generated`
- `verification_failed`
- `user_accepted`

不要在代码里通过隐式 if/else 静默改状态。

### 4.4 自动推进必须满足明确条件

只有在以下条件满足时系统才可自动推进：

- 必要对象已生成
- 当前状态允许推进
- 没有阻塞问题
- 未命中升级条件
- 验证结果满足阈值

### 4.5 所有失败都必须可分类

失败至少分为：

- 可重试失败
- 需返工失败
- 需重规划失败
- 需人工决策失败
- 系统性错误

## 5. 顶层流程状态机

顶层流程用于表示一个需求从接收到阶段性交付的主生命周期。

## 5.1 状态枚举

```text
RECEIVED
CLARIFYING
SPEC_DEFINED
PLANNING
READY_FOR_EXECUTION
EXECUTING
VERIFYING
ACCEPTANCE_READY
WAITING_FOR_USER_ACCEPTANCE
APPROVED
NEEDS_REWORK
NEEDS_REPLAN
PAUSED
CANCELLED
COMPLETED
FAILED
```

## 5.2 状态含义

### RECEIVED
系统已收到原始需求，但尚未进行处理。

### CLARIFYING
系统正在做需求解析、缺失信息补全、默认假设生成和必要澄清。

### SPEC_DEFINED
已生成结构化 spec，且 spec 达到进入规划阶段的最小质量门槛。

### PLANNING
系统正在拆 milestone、拆 tasks、建立依赖、分配角色。

### READY_FOR_EXECUTION
规划已完成，且具备启动执行的条件。

### EXECUTING
OpenCode 执行层正在处理当前批次任务。

### VERIFYING
正在对执行结果进行统一验证，包括构建、测试、审查和验收规则检查。

### ACCEPTANCE_READY
已形成用户可验收成果包，但尚未进入等待用户状态。

### WAITING_FOR_USER_ACCEPTANCE
系统已经暂停自动推进，等待用户审批。

### APPROVED
用户已同意当前阶段结果，可继续推进。

### NEEDS_REWORK
当前结果存在问题，但不需要重做规格或规划，只需要返工。

### NEEDS_REPLAN
当前结果已经不适合继续按原规划推进，需要重新拆分任务或重新安排 milestone。

### PAUSED
项目被显式暂停。

### CANCELLED
项目被终止，不再推进。

### COMPLETED
所有 milestone 均完成并通过验收。

### FAILED
系统进入不可恢复失败状态。

## 5.3 顶层状态迁移图

```text
RECEIVED
-> CLARIFYING
-> SPEC_DEFINED
-> PLANNING
-> READY_FOR_EXECUTION
-> EXECUTING
-> VERIFYING
-> ACCEPTANCE_READY
-> WAITING_FOR_USER_ACCEPTANCE
-> APPROVED
-> READY_FOR_EXECUTION / COMPLETED
```

异常与回退路径：

```text
CLARIFYING -> FAILED
SPEC_DEFINED -> NEEDS_REPLAN
PLANNING -> NEEDS_REPLAN
EXECUTING -> NEEDS_REWORK
VERIFYING -> NEEDS_REWORK / NEEDS_REPLAN / FAILED
WAITING_FOR_USER_ACCEPTANCE -> APPROVED / NEEDS_REWORK / NEEDS_REPLAN / CANCELLED / PAUSED
NEEDS_REWORK -> EXECUTING
NEEDS_REPLAN -> PLANNING
PAUSED -> READY_FOR_EXECUTION / CANCELLED
```

## 5.4 顶层状态迁移表

| 当前状态 | 触发事件 | 下一个状态 | 条件 |
|---|---|---|---|
| RECEIVED | requirement_ingested | CLARIFYING | requirement 对象创建成功 |
| CLARIFYING | clarification_completed | SPEC_DEFINED | open_questions 低于阈值或默认假设可接受 |
| CLARIFYING | clarification_failed | FAILED | 关键输入缺失且无法补全 |
| SPEC_DEFINED | planning_started | PLANNING | spec 校验通过 |
| PLANNING | planning_completed | READY_FOR_EXECUTION | milestone 与 task 生成成功 |
| READY_FOR_EXECUTION | execution_started | EXECUTING | 当前轮任务已派发 |
| EXECUTING | execution_batch_finished | VERIFYING | 当前轮任务均进入已完成或失败可验证状态 |
| VERIFYING | verification_passed | ACCEPTANCE_READY | 当前 milestone 满足验收条件 |
| VERIFYING | verification_failed_retryable | NEEDS_REWORK | 失败可通过局部返工修复 |
| VERIFYING | verification_failed_replan | NEEDS_REPLAN | 原计划不再合理 |
| ACCEPTANCE_READY | acceptance_package_created | WAITING_FOR_USER_ACCEPTANCE | 验收报告已生成 |
| WAITING_FOR_USER_ACCEPTANCE | user_approved | APPROVED | 用户通过 |
| WAITING_FOR_USER_ACCEPTANCE | user_requested_rework | NEEDS_REWORK | 用户要求返工 |
| WAITING_FOR_USER_ACCEPTANCE | user_requested_replan | NEEDS_REPLAN | 用户要求改范围或改方案 |
| WAITING_FOR_USER_ACCEPTANCE | user_paused | PAUSED | 用户暂停 |
| WAITING_FOR_USER_ACCEPTANCE | user_cancelled | CANCELLED | 用户终止 |
| APPROVED | next_milestone_ready | READY_FOR_EXECUTION | 仍有剩余 milestone |
| APPROVED | project_finished | COMPLETED | 全部 milestone 完成 |

## 6. Requirement 状态机

Requirement 表示用户原始需求及其结构化理解过程。

## 6.1 Requirement 状态枚举

```text
RAW
PARSED
CLARIFIED
SPEC_READY
BLOCKED
DROPPED
```

## 6.2 状态说明

### RAW
刚接收的原始输入。

### PARSED
系统已提取目标、约束、用户价值、主要功能点。

### CLARIFIED
缺失信息已被补全或形成可接受默认假设。

### SPEC_READY
已经具备生成 spec 的条件。

### BLOCKED
需求存在关键阻塞，例如目标不清、依赖外部信息、范围冲突。

### DROPPED
该需求被放弃，不再进入后续流程。

## 6.3 迁移规则

```text
RAW -> PARSED -> CLARIFIED -> SPEC_READY
RAW / PARSED / CLARIFIED -> BLOCKED
BLOCKED -> CLARIFIED
BLOCKED -> DROPPED
```

## 7. Milestone 状态机

Milestone 表示一个阶段性交付目标，是用户验收的主要单位。

## 7.1 Milestone 状态枚举

```text
PLANNED
READY
IN_PROGRESS
VERIFYING
READY_FOR_ACCEPTANCE
ACCEPTED
REWORK_REQUIRED
REPLANNING_REQUIRED
SKIPPED
CANCELLED
```

## 7.2 状态说明

### PLANNED
Milestone 已定义，但尚未准备开始。

### READY
Milestone 所需前置依赖已满足，可进入执行。

### IN_PROGRESS
对应任务已开始执行。

### VERIFYING
当前 milestone 的任务结果正在做统一校验。

### READY_FOR_ACCEPTANCE
已形成 milestone 验收包。

### ACCEPTED
该 milestone 已通过用户验收。

### REWORK_REQUIRED
无需改 milestone 结构，但需要返工。

### REPLANNING_REQUIRED
原 milestone 范围、拆法或依赖不再合适。

### SKIPPED
该 milestone 被跳过，不再执行。

### CANCELLED
该 milestone 被取消。

## 7.3 迁移规则

```text
PLANNED -> READY
READY -> IN_PROGRESS
IN_PROGRESS -> VERIFYING
VERIFYING -> READY_FOR_ACCEPTANCE
VERIFYING -> REWORK_REQUIRED
VERIFYING -> REPLANNING_REQUIRED
READY_FOR_ACCEPTANCE -> ACCEPTED
READY_FOR_ACCEPTANCE -> REWORK_REQUIRED
READY_FOR_ACCEPTANCE -> REPLANNING_REQUIRED
REWORK_REQUIRED -> IN_PROGRESS
REPLANNING_REQUIRED -> PLANNED
```

## 8. Task 状态机

Task 是最小执行单元，是调度和返工的基本粒度。

## 8.1 Task 状态枚举

```text
TODO
ASSIGNED
IN_PROGRESS
IMPLEMENTED
TESTED
REVIEWED
VERIFIED
DONE
BLOCKED
FAILED_TEST
CHANGES_REQUESTED
FAILED_VERIFICATION
ESCALATED
CANCELLED
```

## 8.2 状态说明

### TODO
任务已生成但未分派。

### ASSIGNED
已选择 owner_role，等待执行。

### IN_PROGRESS
执行中。

### IMPLEMENTED
实现完成，等待测试或下游处理。

### TESTED
测试任务已执行完毕。

### REVIEWED
审查任务已执行完毕。

### VERIFIED
通过统一验证。

### DONE
任务闭环结束。

### BLOCKED
存在阻塞，无法继续。

### FAILED_TEST
测试未通过。

### CHANGES_REQUESTED
review 未通过，要求修改。

### FAILED_VERIFICATION
统一验证阶段失败。

### ESCALATED
任务问题超出自动修复范围，已升级。

### CANCELLED
任务被取消。

## 8.3 典型任务迁移流

### 开发任务

```text
TODO
-> ASSIGNED
-> IN_PROGRESS
-> IMPLEMENTED
-> REVIEWED
-> VERIFIED
-> DONE
```

### 开发 + 测试任务链

```text
IMPLEMENTED -> TESTED -> REVIEWED -> VERIFIED -> DONE
```

### 失败返工流

```text
IN_PROGRESS -> BLOCKED
IMPLEMENTED -> FAILED_TEST
REVIEWED -> CHANGES_REQUESTED
VERIFIED -> FAILED_VERIFICATION
FAILED_TEST -> IN_PROGRESS
CHANGES_REQUESTED -> IN_PROGRESS
FAILED_VERIFICATION -> IN_PROGRESS
```

### 升级流

```text
BLOCKED -> ESCALATED
FAILED_TEST -> ESCALATED
CHANGES_REQUESTED -> ESCALATED
FAILED_VERIFICATION -> ESCALATED
```

## 8.4 自动推进规则

一个 task 只有满足以下条件才允许自动进入下一状态：

- 当前状态允许推进
- 上一状态产物存在
- 所需依赖任务已完成
- retry_count 未超阈值
- 未触发高风险中断条件

## 10. 事件模型设计

状态机的驱动核心是事件。

## 10.1 事件对象结构

```yaml
event_id:
event_type:
entity_type:
entity_id:
timestamp:
triggered_by:
source:
payload:
correlation_id:
causation_id:
```

## 10.2 字段说明

- `event_id`: 事件唯一 ID
- `event_type`: 事件类型
- `entity_type`: 作用对象类型，如 project / requirement / milestone / task / acceptance
- `entity_id`: 对象 ID
- `timestamp`: 事件时间
- `triggered_by`: 触发者，如 system / user / agent / verifier
- `source`: 事件来源模块
- `payload`: 附加数据
- `correlation_id`: 用于串联同一轮流程
- `causation_id`: 上一个导致该事件的事件 ID

## 10.3 推荐事件类型

### Requirement 相关
- requirement_ingested
- requirement_parsed
- requirement_blocked
- clarification_completed
- clarification_failed

### Spec 相关
- spec_generation_started
- spec_generated
- spec_rejected
- spec_approved

### Planning 相关
- planning_started
- planning_completed
- milestone_created
- task_created
- dispatch_completed

### Execution 相关
- task_assigned
- task_started
- task_output_generated
- task_blocked
- task_failed
- task_completed

### Verification 相关
- verification_started
- verification_passed
- verification_failed_retryable
- verification_failed_replan
- verification_failed_fatal

### Acceptance 相关
- acceptance_package_created
- user_approved
- user_requested_rework
- user_requested_replan
- user_rejected
- project_completed

```yaml
id:

```

## 13.2 关键字段说明

### raw_input
用户原始输入文本，必须保留。

### normalized_input
解析后的结构化描述，可作为 spec 输入。

### business_goal
该需求想解决的业务目标。

### user_value
对最终用户或项目方的价值。

### scope_hint
范围提示，不要求完全准确，但应给出大致边界。

### assumptions
系统自动补充的默认假设。

### open_questions
尚未解决的问题列表。

### acceptance_hint
对后续验收标准的提示性描述。

## 14. Spec 对象设计

Spec 是从 requirement 衍生出的结构化规格对象，是整个规划阶段的输入。

## 14.1 Spec 结构

```yaml
id:
project_id:
requirement_id:
version:
status:
summary:
background:
goals:
scope:
out_of_scope:
functional_requirements:
non_functional_requirements:
acceptance_criteria:
technical_constraints:
assumptions:
risks:
dependencies:
created_at:
updated_at:
approved_by_user:
```

## 14.2 字段约束

- `summary` 必须有
- `scope` 必须有
- `out_of_scope` 必须有
- `functional_requirements` 至少 1 项
- `acceptance_criteria` 至少 1 项
- `approved_by_user` 在 MVP 中可为空，表示系统先继续；进入强验收节点时再要求用户确认

## 16. Task 对象设计

Task 是核心执行对象。

## 16.1 Task 结构

```yaml

```

## 16.2 字段说明

| 字段 | 说明 |
|---|---|
| parent_task_id | 可选，用于子任务拆分 |
| type | 任务类型，如 architecture / implementation / test / review / docs |
| owner_role | 默认执行角色 |
| inputs | 输入工件引用或内联上下文 |
| constraints | 本任务额外约束 |
| expected_outputs | 输出要求 |
| dependencies | 依赖 task 列表 |
| artifacts | 产出工件 ID 列表 |
| verification_steps | 验证方法列表 |
| retry_count | 当前重试次数 |
| max_retry | 最大自动重试次数 |
| risk_level | low / medium / high / critical |
| blocking_reason | 阻塞原因 |

## 21. 对象关系约束

## 21.1 一对多关系

- 一个 Project 可有多个 Requirement
- 一个 Requirement 可派生多个 Spec 版本
- 一个 Spec 可对应多个 Milestone
- 一个 Milestone 可包含多个 Task
- 一个 Task 可产生多个 Artifact
- 一个 Milestone 可有多个 VerificationResult
- 一个 Milestone 可有多个 AcceptanceReport 版本

## 21.2 一对一约束

MVP 阶段建议：

- 同一时刻一个 Project 只允许有一个 active_spec
- 同一时刻一个 Milestone 只允许一个 active acceptance 流程
- 一个 Task 在任一时刻只允许一个主 owner_role

## 21.3 依赖约束

- Task 不能依赖不存在的 task
- 不允许形成 task dependency 环
- 未满足 dependencies 的 task 不可自动进入 `IN_PROGRESS`

## 22. 对象版本化策略

为避免覆盖历史，建议做轻量版本化。

### 22.1 需要版本化的对象
- Requirement
- Spec
- AcceptanceReport

### 22.2 版本化规则
- 修改后创建新版本，不覆盖原文件
- 当前版本通过 `active_*_id` 指向
- 版本号采用 `v1`, `v2`, `v3` 即可

### 22.3 不强制版本化的对象
- Task
- VerificationResult
- EventLog

这些对象更适合追加记录。

## 25. 自动推进判定逻辑

## 25.1 顶层自动推进判定

伪规则：

```text
if current_state in auto_progress_allowed
and required_artifacts_exist
and blocking_items == 0
and retry_limit_not_exceeded
and escalation_not_triggered
then move_to(next_state)
```

## 25.2 Task 自动推进判定

```text
TODO -> ASSIGNED
条件：owner_role 已确定

ASSIGNED -> IN_PROGRESS
条件：dependencies 全部 DONE，且执行资源可用

IN_PROGRESS -> IMPLEMENTED
条件：执行层输出满足 expected_outputs 最小要求

IMPLEMENTED -> TESTED / REVIEWED
条件：下游测试或审查任务已完成

REVIEWED -> VERIFIED
条件：review 通过或 warning 在可接受范围内

VERIFIED -> DONE
条件：verification status = PASS 或 PASS_WITH_WARNINGS
```

## 26. 升级与中断规则

## 26.1 必须升级的情况

以下情况不允许继续自动推进：

- requirement 中存在关键业务歧义
- spec 的 scope 与 goals 明显冲突
- milestone 依赖无法满足
- task 连续失败超过 max_retry
- 触达 critical 风险等级模块
- 用户明确要求确认的决策点
- verification 返回 FAIL_ESCALATE
- 用户主动暂停或终止

## 26.2 升级对象

升级可分为两种：

### 升级到管理层
适合：
- 局部任务失败
- task 依赖需要重排
- 需要改派角色

### 升级到用户
适合：
- 目标变化
- 范围变更
- 架构 trade-off 无法自动决策
- 当前里程碑是否还值得继续
