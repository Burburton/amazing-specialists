# Anti-Example: Unrealistic Performance Targets

## 错误做法

设定不切实际的性能目标，无法实现或验证。

## 问题代码/配置

```yaml
# 错误示例
performance_requirements:
  # 问题1: 技术上不可行
  response_time: < 1ms  # 网络延迟都不止1ms
  
  # 问题2: 无限并发
  concurrent_users: 10000000  # 单机不可能支撑
  
  # 问题3: 零错误
  error_rate: 0%  # 分布式系统不可能零错误
  
  # 问题4: 无限吞吐
  throughput: ∞ RPS
```

## 为什么这是错的

1. **物理限制**: 网络延迟、磁盘 I/O 有物理下限
2. **资源限制**: 单机 CPU、内存有限
3. **分布式现实**: 网络分区、超时不可避免
4. **成本效益**: 过高目标成本不成比例

## 正确做法

```yaml
# 正确示例
performance_requirements:
  # 1. 基于技术可行性
  response_time:
    p50: < 50ms   # 大多数请求
    p99: < 200ms  # 尾部延迟
    
  # 2. 基于资源规划
  concurrent_users: 5000  # 基于服务器容量规划
  # 如需更多，考虑水平扩展
  
  # 3. 合理的错误容忍
  error_rate: < 0.1%  # 可接受的错误率
  
  # 4. 可达成的吞吐量
  throughput: 
    target: 2000 RPS   # 当前架构目标
    max: 5000 RPS      # 扩容后可达
```

## 设定目标的步骤

1. **了解限制**: 网络、硬件、软件限制
2. **参考基线**: 历史性能数据
3. **业务需求**: 实际业务场景需求
4. **迭代调整**: 根据测试结果调整

## 判断目标是否合理的标准

| 指标 | 合理范围 | 说明 |
|------|----------|------|
| P99 响应时间 | 50-500ms | 视接口复杂度 |
| 并发用户 | 架构容量 50-80% | 留出 buffer |
| 错误率 | 0.01%-1% | 视业务容忍度 |
| 吞吐量 | 基线 80-120% | 渐进增长 |
