# 角色质量保障规范

## 1. 文档目的

本文档用于定义“全自动产品研发闭环 MVP”中的 **角色质量保障规范**，为 OpenClaw 管理层与 OpenCode 执行层之间的协作提供统一、可执行、可验证的质量标准。

本文档解决的问题包括：

- 如何定义各角色的高质量交付
- 如何保证不同角色的输出可被下游稳定消费
- 如何防止角色越权、漏项、瞎扩范围
- 如何通过 gate、checklist、artifact、verification、rework 形成质量闭环
- 如何将“高质量”从抽象要求落实为可执行规范

本文档适用于以下角色：

- architect
- developer
- tester
- reviewer
- docs
- security
- performance（可选增强）
- release（可选增强）

---

## 2. 适用范围

本规范适用于：

- 所有由 OpenClaw 派发给 OpenCode 专家角色的 task
- 所有进入 milestone 闭环的设计、实现、测试、审查、文档与专项检查产物
- 所有需要进入 verification / acceptance / rework / escalation 的执行结果

本规范不直接规定：

- 商业优先级选择
- 产品方向拍板
- 用户侧验收偏好
- 多项目资源仲裁

这些由管理层或用户决策承担。

---

## 3. 核心原则

## 3.1 角色清晰

每个角色只对其主职责负责，禁止默认承担其他角色的主要职责。

例如：
- architect 不主导大规模业务代码实现
- developer 不负责最终验收拍板
- tester 不替代 reviewer 做架构放行
- reviewer 不偷偷扩大实现范围
- docs 不改业务逻辑
- security 不代替开发完成普通功能闭环

## 3.2 输入结构化

所有角色必须接收结构化 dispatch payload，不允许只根据模糊自然语言直接执行关键任务。

## 3.3 输出标准化

所有角色必须输出统一格式的 execution result 与 artifact，禁止仅返回“已完成”“没问题”“看起来可以”之类不可消费结论。

## 3.4 独立验证

任何角色都不能自证质量。  
角色输出只代表“候选结果”，真正的质量结论必须通过 verification / acceptance 层裁定。

## 3.5 最小改动与最小返工

所有角色默认遵守最小范围原则：

- 不擅自扩大改动边界
- 不擅自补做未授权目标
- 失败时优先局部修复，不轻易推翻整轮产出

## 3.6 显式暴露问题

角色遇到以下情况必须显式输出，而非隐式忽略：

- 缺失上下文
- 范围冲突
- 风险超阈值
- 无法完成的部分
- 未验证的部分
- 建议升级的部分

---

## 4. 质量保障体系总览

角色质量保障由 8 个层面组成：

1. 角色边界
2. 输入契约
3. 输出契约
4. 专属 skill 方法包
5. 角色 gate
6. 独立 verification
7. 返工与升级规则
8. 质量度量指标

整体闭环如下：

```text
Dispatch Payload
-> Role Execution
-> Standard Artifact
-> Role Gate Check
-> Verification
-> Acceptance
-> Rework / Replan / Escalate / Continue
```

---

## 5. 统一输入契约

所有角色都必须基于统一 dispatch payload 工作。

## 5.1 必填字段

```yaml
dispatch_id:
project_id:
milestone_id:
task_id:
role:
command:
title:
goal:
description:
context:
constraints:
inputs:
expected_outputs:
verification_steps:
risk_level:
```

## 5.2 可选字段

```yaml
retry_context:
upstream_dependencies:
downstream_expectations:
metadata:
```

## 5.3 输入质量最低要求

若缺少以下任一项，角色应优先返回 `BLOCKED` 或 `FAILED_ESCALATE`，而不是盲目执行：

- 当前任务目标不清
- 当前任务范围不清
- 关键约束缺失
- 关键输入工件缺失
- role 与 command 明显不匹配

## 5.4 输入处理要求

角色必须：

- 明确读取 `goal`
- 明确读取 `constraints`
- 明确识别 `expected_outputs`
- 明确识别 `verification_steps`
- 优先使用 `inputs` 与 artifact，而不是自行臆测项目状态

---

## 6. 统一输出契约

所有角色必须输出统一 execution result。

## 6.1 标准输出结构

```yaml
dispatch_id:
project_id:
milestone_id:
task_id:
role:
command:
status:
summary:
artifacts:
changed_files:
checks_performed:
issues_found:
risks:
recommendation:
needs_followup:
followup_suggestions:
escalation:
metadata:
```

## 6.2 status 枚举

```text
SUCCESS
SUCCESS_WITH_WARNINGS
PARTIAL
BLOCKED
FAILED_RETRYABLE
FAILED_ESCALATE
```

## 6.3 输出要求

### summary
必须简要说明：
- 做了什么
- 是否达成目标
- 还缺什么

### artifacts
必须列出本轮主要产物引用。  
如无 artifact，必须说明原因。

### changed_files
有文件改动时必须列出。  
无改动也必须明确写无改动。

### checks_performed
必须说明本角色已经做了哪些自检，不允许暗示未做的验证已完成。

### issues_found
必须列出发现但未解决的问题。

### risks
必须列出当前剩余风险；若无，明确写“未发现明显新增风险”。

### recommendation
必须明确建议后续流向，例如：
- CONTINUE
- SEND_TO_TEST
- SEND_TO_REVIEW
- REWORK
- REPLAN
- ESCALATE

---

## 7. 统一 artifact 规范

## 7.1 artifact 最小结构

```yaml
artifact_id:
artifact_type:
title:
path:
format:
summary:
created_by_role:
related_task_id:
created_at:
metadata:
```

## 7.2 artifact 基本要求

- artifact 必须可引用
- artifact 标题必须能表达内容
- artifact summary 必须简要准确
- 不允许只说“见上文”“见输出”
- 不允许 artifact 缺少类型或关联 task

## 7.3 推荐 artifact_type

- design_note
- implementation_summary
- code_diff_summary
- test_report
- review_report
- doc_update_report
- security_report
- performance_report
- release_readiness_report

---

## 8. 通用质量要求

所有角色都必须满足以下通用要求。

## 8.1 不得超范围执行

角色不得在未授权的情况下：

- 擅自扩大改动范围
- 顺手做额外需求
- 擅自改变 milestone 目标
- 擅自引入新依赖
- 擅自重构不相关模块

如确需扩展，必须显式提出。

## 8.2 不得伪造完成

禁止：

- 未运行测试却声称测试通过
- 未审查风险却声称无风险
- 未同步文档却声称文档已更新
- 未检查边界却声称边界覆盖完整

## 8.3 必须暴露局限

角色必须显式说明：
- 哪些部分未验证
- 哪些部分缺少上下文
- 哪些判断是基于假设
- 哪些问题需要下游角色继续检查

## 8.4 必须产出可消费结果

输出必须让下游角色或管理层可以继续工作。  
否则视为低质量交付。

---

## 9. 角色边界规范

## 9.1 architect 边界

### 负责
- 技术方案
- 模块边界
- 接口契约
- 风险识别
- 实施顺序建议

### 不负责
- 大规模业务代码实现
- 最终测试闭环
- 最终验收拍板

### 允许的少量代码
- 示例代码
- 伪代码
- 接口草图
- 配置片段

### 禁止
- 未经授权直接完成整个 feature 实现

---

## 9.2 developer 边界

### 负责
- 功能实现
- bug 修复
- 局部重构
- 必要脚本 / 配置补齐
- 自检

### 不负责
- 重新定义需求
- 自行决定 milestone 通过
- 跳过测试与 review 直接放行

### 禁止
- 擅自扩大范围
- 以“顺手优化”为由做不相关重构

---

## 9.3 tester 边界

### 负责
- 测试设计
- 测试执行
- 回归分析
- 边界覆盖
- 失败分类

### 不负责
- 大规模业务功能实现
- 架构正确性最终裁定
- 产品范围取舍

### 禁止
- 通过修改业务逻辑掩盖测试失败

---

## 9.4 reviewer 边界

### 负责
- 独立审查
- spec 与实现比对
- 风险审查
- 放行 / 拒绝建议

### 不负责
- 主导实现
- 大规模补代码
- 替代 tester 做验证闭环

### 禁止
- 一边 review 一边偷偷完成实现，导致角色失真

---

## 9.5 docs 边界

### 负责
- README 同步
- 技术文档同步
- changelog
- 用户说明更新

### 不负责
- 改业务逻辑
- 决定功能是否通过验收

---

## 9.6 security 边界

### 负责
- 认证 / 权限检查
- 输入校验检查
- secret / token / credential 处理检查
- 高风险依赖检查
- 安全 gate 建议

### 不负责
- 普通功能闭环实现
- 产品 trade-off 取舍

---

## 10. 角色通用 gate

在进入 verification 前，所有角色必须先过自己的 role gate。

## 10.1 role gate 的意义

role gate 不是最终验收，而是角色自身交付的最低质量门槛。  
没过 gate 的结果，不应直接进入下游。

## 10.2 通用 gate 检查项

所有角色都必须通过以下检查：

- 输入理解是否完整
- 输出是否满足 expected_outputs
- 是否有标准 artifact
- 是否清楚说明未完成项
- 是否清楚说明风险
- 是否明确 recommendation
- 是否未超出 constraints

若任意核心项缺失，则默认判为低质量结果。

---

## 11. architect 质量规范

## 11.1 交付定义

architect 的高质量交付应满足：

- 能把 requirement / spec 转为可执行设计
- 设计边界清晰
- 接口契约清晰
- 风险清晰
- 后续 developer 能据此实现

## 11.2 必备输出

必须包含：
- design summary
- module boundary
- interface / data contract
- assumptions
- risks
- recommended implementation order

## 11.3 gate 检查项

architect 输出必须检查：

- 是否明确当前 task goal
- 是否明确 scope / out_of_scope
- 是否明确模块边界
- 是否明确接口输入输出
- 是否明确依赖与风险
- 是否可直接交给 developer

## 11.4 常见低质量模式

- 方案过于抽象，无法落地
- 只写原则，不写边界
- 不写 assumptions
- 不写 trade-off
- 风险未暴露
- implementation steps 不可执行

## 11.5 architect 专属质量指标

- 设计被 developer 直接采用率
- 因设计不清导致的 rework 率
- architecture task 一次通过率
- design note 缺失关键信息率

---

## 12. developer 质量规范

## 12.1 交付定义

developer 的高质量交付应满足：

- 正确实现 task goal
- 不超范围改动
- 输出完整实现总结
- 对未完成项与风险透明
- 具备进入 tester / reviewer 的条件

## 12.2 必备输出

必须包含：
- changed files
- implementation summary
- deviations from design（如有）
- self-check result
- unresolved issues

## 12.3 gate 检查项

developer 输出必须检查：

- 是否满足 expected_outputs
- 是否遵守 constraints
- 是否列出 changed files
- 是否说明实现范围
- 是否说明未解问题
- 是否说明已做哪些自检
- 是否未隐瞒风险

## 12.4 自检最低要求

developer 至少应做并显式记录：

- 实现目标对齐检查
- 改动范围检查
- 依赖引入检查
- 关键路径自读检查
- 与 design note 一致性检查（若有）

## 12.5 常见低质量模式

- 代码写了，但无总结
- 改了不相关文件
- 与 design 偏离但未说明
- 自检缺失
- 失败路径没处理
- 假装测试已经通过

## 12.6 developer 专属质量指标

- implementation task 一次通过率
- 因超范围改动导致的 review reject 率
- 因遗漏边界条件导致的 test fail 率
- 自检缺失率

---

## 13. tester 质量规范

## 13.1 交付定义

tester 的高质量交付应满足：

- 能证明实现是否满足关键路径
- 能指出测试覆盖与缺口
- 能清晰分类失败
- 能给出可执行回归结论

## 13.2 必备输出

必须包含：
- test scope
- tests added / tests run
- pass/fail summary
- uncovered gaps
- edge cases checked
- retryable / non-retryable 判断

## 13.3 gate 检查项

tester 输出必须检查：

- 是否覆盖 acceptance criteria 的关键路径
- 是否说明测试范围
- 是否明确 pass/fail
- 是否给出 gap 分析
- 是否把 failure 分类清楚
- 是否避免用“感觉没问题”替代测试结论

## 13.4 常见低质量模式

- 只跑 happy path
- 不写 coverage gap
- 失败不分类
- 不区分 test design 和 test execution
- 无法复现失败原因

## 13.5 tester 专属质量指标

- 测试报告完整率
- regression 缺口率
- 可重试失败识别准确率
- 关键路径覆盖率

---

## 14. reviewer 质量规范

## 14.1 交付定义

reviewer 的高质量交付应满足：

- 独立判断实现是否可放行
- 能识别 must-fix 与非阻塞问题
- 能把 spec / design / implementation 之间的不一致指出来
- 给出可执行修改建议

## 14.2 必备输出

必须包含：
- overall decision
- must-fix issues
- non-blocking issues
- residual risks
- actionable suggestions

## 14.3 gate 检查项

reviewer 输出必须检查：

- 是否有明确 approve / reject / warn 结论
- 是否区分 must-fix 和建议项
- 是否说明风险
- 是否说明为何拒绝或通过
- 是否给出可执行 action items

## 14.4 常见低质量模式

- 只说“需要优化”但不说怎么改
- 不区分严重性
- 不对齐 spec / design
- 只看代码风格，不看目标达成
- 偷偷自己补实现

## 14.5 reviewer 专属质量指标

- review report 可执行性评分
- must-fix 漏检率
- review reject 后一次修复通过率
- spec / implementation mismatch 发现率

---

## 15. docs 质量规范

## 15.1 交付定义

docs 的高质量交付应满足：

- 能把用户侧或开发侧需要知道的变更同步到文档
- 不夸大已完成情况
- 能指出文档缺口

## 15.2 必备输出

必须包含：
- synced docs list
- missing docs
- user-facing summary
- internal summary（如适用）

## 15.3 gate 检查项

docs 输出必须检查：

- 是否明确更新了哪些文档
- 是否说明没覆盖的文档项
- 是否与实现结果对齐
- 是否避免写出未实现功能

## 15.4 常见低质量模式

- changelog 空泛
- README 与实际不符
- 文档遗漏风险说明
- 文档提前描述未完成功能

## 15.5 docs 专属质量指标

- 文档同步完整率
- 文档与实现偏差率
- 用户侧关键说明缺失率

---

## 16. security 质量规范

## 16.1 交付定义

security 的高质量交付应满足：

- 能识别当前改动中的安全风险
- 能按严重级别分类
- 能明确哪些问题阻断放行
- 能给出安全 gate 建议

## 16.2 必备输出

必须包含：
- scope checked
- issues by severity
- must-fix list
- residual risks
- gate recommendation

## 16.3 gate 检查项

security 输出必须检查：

- 是否明确检查范围
- 是否明确 high / critical issue
- 是否区分 must-fix 与可延期项
- 是否给出放行建议
- 是否避免泛泛而谈

## 16.4 常见低质量模式

- 只给“注意安全”类空话
- 不区分严重性
- 不说明检查范围
- 未识别权限 / secret / 输入校验类问题

## 16.5 security 专属质量指标

- 高风险问题发现率
- severity 分类准确率
- 安全报告完整率
- 安全 gate 阻断有效率

---

## 17. performance 质量规范（可选）

## 17.1 交付定义

performance 的高质量交付应满足：

- 能识别热点与回归
- 能说明问题影响面
- 能区分必须修复与可观察项

## 17.2 必备输出

必须包含：
- checked scope
- hotspot summary
- regression evidence
- impact assessment
- recommendation

---

## 18. release 质量规范（可选）

## 18.1 交付定义

release 的高质量交付应满足：

- 能说明当前版本是否具备进入发布候选的条件
- 能汇总风险与阻断项
- 能生成清晰发布说明

## 18.2 必备输出

必须包含：
- readiness summary
- blockers
- known risks
- release note draft
- go / no-go recommendation

---

## 19. 统一 skill 规范

## 19.1 每个 skill 必须包含的部分

```md
# Skill Name

## Purpose
## When to Use
## When Not to Use
## Inputs
## Steps
## Checklists
## Common Failure Modes
## Output Requirements
## Examples
```

## 19.2 skill 质量要求

一个高质量 skill 应满足：

- 适用场景明确
- 步骤可执行
- checklist 可检查
- failure modes 真实且常见
- 输出要求能被下游消费

## 19.3 skill 不应退化为

- 纯概念说明书
- 大段无结构知识堆砌
- 含糊 prompt 杂糅
- 与角色边界冲突的方法说明

---

## 20. command 质量规范

## 20.1 command 作用

command 用于固化动作入口，减少 prompt 随机性。

## 20.2 command 要求

每个 command 必须明确：

- 适用角色
- 适用场景
- 所需输入
- 预期输出
- 禁止行为
- 后续推荐流向

## 20.3 command 不应过多

第一版建议每个角色 2 到 3 个 command。  
太多会让维护复杂化，太少则动作不稳定。

---

## 21. 质量 gate 体系

## 21.1 gate 分层

建议设置以下 gate：

- role gate
- build gate
- test gate
- review gate
- security gate
- docs gate
- acceptance gate

## 21.2 角色与 gate 映射

| 角色 | 必经 gate |
|---|---|
| architect | role gate, acceptance gate |
| developer | role gate, build gate, review gate |
| tester | role gate, acceptance gate |
| reviewer | role gate, acceptance gate |
| docs | role gate, docs gate |
| security | role gate, security gate |

## 21.3 gate 阻断规则

任何 must-pass gate 未通过时，不得自动推进到下一阶段。

## 22. 失败与返工规范

## 22.1 失败分类

推荐统一使用以下分类：

- missing_artifact
- incomplete_output
- invalid_format
- build_failed
- test_failed
- review_rejected
- docs_missing
- security_issue
- perf_regression
- invalid_scope
- dependency_conflict
- ambiguous_goal
- unresolved_tradeoff

## 22.2 返工规则

返工必须：

- 显式创建 ReworkRequest
- 指明 failed checks
- 指明 required fixes
- 指明 non-goals
- 保留 retry_count
- 附带相关 logs / comments / artifact

## 22.3 升级规则

以下情况必须升级：

- retry_count 超限
- high / critical security issue
- role 无法判断 trade-off
- 范围与目标冲突
- 当前结果是否继续需要用户拍板

## 24. 质量审计建议

为了长期稳定，建议定期做角色质量审计。

## 24.1 审计频率建议

- MVP 阶段：每个 milestone 审计一次
- 稳定后：每 2 到 4 个 milestone 一次

## 24.2 审计内容

- 哪类 task 最常失败
- 哪个角色最常输出不可消费结果
- 哪类 artifact 最容易缺失
- 哪个 gate 最常阻断
- 哪类返工最常重复出现

## 24.3 审计目标

不是责备某个角色，而是：

- 优化 skill
- 优化 command
- 优化 dispatch payload
- 优化 templates
- 优化 gate

---

## 25. 推荐的落地方式

## 25.1 先落这 5 件事

第一优先级：

1. 统一 dispatch payload schema
2. 统一 execution result schema
3. 每角色一个最小输出模板
4. 每角色一个 role gate checklist
5. 统一 ReworkRequest / Escalation schema

## 25.2 再落 skill 和 command

第二优先级：

- architect 3 个 skill
- developer 3 个 skill
- tester 3 个 skill
- reviewer 3 个 skill

每个角色先做 2 个 command 即可。

## 25.3 最后做质量审计

第三优先级：

- 指标采集
- milestone 质量报告
- 角色质量回顾

---

## 26. 最小执行清单

如果你要把这份规范真正用起来，最低要做到：

- [ ] 所有角色使用统一输入 payload
- [ ] 所有角色输出统一 result
- [ ] 所有角色产出标准 artifact
- [ ] architect / developer / tester / reviewer 都有 role gate
- [ ] 所有失败都进入统一分类
- [ ] 返工通过 ReworkRequest 驱动
- [ ] 高风险问题进入 Escalation
- [ ] milestone 统一产出 AcceptanceReport

---

## 27. 验收标准

本规范在实现层应满足以下标准：

1. 每个角色有明确边界
2. 每个角色有最低输出要求
3. 每个角色有 role gate
4. 所有角色使用统一输入输出契约
5. 所有角色输出都能被下游稳定消费
6. 失败能被统一分类
7. 返工与升级规则明确
8. 至少支持 architect / developer / tester / reviewer 四个核心角色
9. 可通过指标度量交付质量
10. 可直接作为 repo 中的执行规范文档使用

---

## 28. 一句话总结

角色 agent 的高质量交付，不靠“它看起来像专家”，而靠：

**清晰边界、结构化输入、标准化输出、专属方法包、分层 gate、独立验证、有限返工、持续审计。**

这 8 件事同时成立，质量才会真正稳定。
