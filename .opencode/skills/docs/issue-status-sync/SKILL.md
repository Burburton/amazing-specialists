# Skill: issue-status-sync

## Purpose

同步任务执行状态到 GitHub Issue，发布进度评论，**不关闭 Issue**（等待管理层验收）。

解决的核心问题：
- **过早关闭 Issue**：developer 完成实现后直接关闭 Issue，跳过测试、审查、验收环节
- **缺少 Issue 状态同步**：执行层完成后没有向 Issue 报告进度
- **验收流程断裂**：Issue 关闭时机与验收决策脱节

## When to Use

必须使用时：
- 任务通过 GitHub Issue 触发（dispatch context 包含 `issue_number`）
- 所有执行角色完成工作（收到 reviewer 的 `acceptance-decision-record` RC-003）
- 需要向 Issue 报告执行结果

推荐使用时：
- 执行完成后需要留存 Issue 进度记录
- 需要通知管理层进行验收

## When Not to Use

不适用场景：
- 任务通过 CLI 触发（无 Issue 关联）
- 任务通过 OpenClaw 触发（由 OpenClaw adapter 处理）
- 执行结果建议 REWORK（Issue 保持开放，等待返工）
- 执行结果建议 ESCALATE（Issue 保持开放，需要人工决策）
- 无 GitHub Issue 上下文

## Business Rules

### BR-001: Evidence Consumption (证据消费)
Issue 评论必须基于上游角色产生的 artifacts：
- `acceptance-decision-record` (RC-003) - reviewer 验收建议
- `execution result` - 执行结果汇总
- `verification-report` (TC-001) - 测试结果

**禁止**：凭空捏造执行状态，不基于 artifact 发布评论。

### BR-002: Minimal Surface (最小表面积)
仅更新 Issue 评论，**不修改**：
- Issue 标题
- Issue 标签
- Issue 状态（state: open/closed）
- Issue 分配人

### BR-003: No Premature Closure (禁止过早关闭)
**严禁**在本 skill 中关闭 Issue。

Issue 关闭的正确时机：
- 管理层（OpenClaw 或人工）验收通过后
- 由 acceptance 层或 management 层执行关闭

### BR-004: Issue Context Check (Issue 上下文检查)
执行前必须验证：
- `dispatch.issue_number` 存在
- `dispatch.repository` 存在
- GitHub API 可访问

若任一条件不满足，跳过本 skill，记录 warning。

### BR-005: Recommendation Mapping (建议映射)
根据 execution result 的 recommendation 决定评论内容：

| Recommendation | 评论内容 | Issue 状态 |
|----------------|----------|-----------|
| CONTINUE / SEND_TO_ACCEPTANCE | "执行完成，等待验收" | 保持 OPEN |
| REWORK | "需要返工，详见..." | 保持 OPEN |
| ESCALATE | "需要人工决策，详见..." | 保持 OPEN |

## Implementation Process

### Phase 1: Context Gathering (上下文收集)

#### Step 1: 验证 Issue 上下文
```
检查 dispatch context:
- issue_number: 存在？
- repository: 存在？
- GitHub API: 可访问？

若任一缺失 → 记录 warning，跳过 skill
```

#### Step 2: 收集上游 Artifacts
使用 `artifact-reading` skill 读取：
1. **acceptance-decision-record** (RC-003)
   - 来源：reviewer 角色
   - 关键字段：decision, must_fix_issues, recommendation
2. **execution result**
   - 来源：common/execution-reporting
   - 关键字段：status, artifacts, recommendation
3. **verification-report** (TC-001) - 可选
   - 来源：tester 角色
   - 关键字段：tests_passed, coverage_summary

#### Step 3: 确定评论类型
基于收集的信息，确定评论模板：
- 成功完成 → "执行完成，等待验收"
- 部分完成 → "部分完成，有已知问题"
- 需要返工 → "需要返工"
- 需要升级 → "需要人工决策"

### Phase 2: Report Generation (报告生成)

#### Step 4: 生成 issue-progress-report (DOC-003)
```yaml
issue_progress_report:
  issue_number: number
  repository: string
  dispatch_id: string
  execution_status: SUCCESS | PARTIAL | FAILED | BLOCKED
  roles_completed:
    - role: architect | developer | tester | reviewer | docs | security
      status: COMPLETE | PARTIAL | SKIPPED
      key_output: string
  artifacts_produced:
    - contract_id: string
      file_path: string
      summary: string
  recommendation: PENDING_ACCEPTANCE | NEEDS_REWORK | BLOCKED_ESCALATION
  recommendation_reason: string
  quality_summary:
    tests_passed: boolean | null
    review_status: APPROVED | REJECTED | WARN | null
    security_gate: PASSED | FAILED | null
  next_steps:
    - action: string
      responsible_party: string
  posted_at: ISO8601
  comment_url: string
```

#### Step 5: 格式化 Markdown 评论
使用 `adapters/github-issue/comment-templates.js` 生成格式化评论：

```markdown

| tester | ✅ Complete | verification-report |

- ✅ 审查通过

### 下一步
```

### Phase 3: Comment Posting (评论发布)

#### Step 6: 调用 GitHub API
```javascript
// 使用 adapters/github-issue/github-client.js
await githubClient.postComment(owner, repo, issueNumber, commentBody);
```

#### Step 7: 验证评论发布
- 检查 API 返回状态码 (201 Created)
- 记录 comment_url
- 不关闭 Issue（BR-003）

### Phase 4: Verification (验证)

#### Step 8: 确认 Issue 状态
```
验证 Issue 状态仍为 OPEN
若已关闭 → 记录 warning（可能被其他流程关闭）
```

#### Step 9: 记录执行结果
```yaml
issue_status_sync_result:
  issue_number: number
  comment_posted: boolean
  comment_url: string
  issue_state_after: OPEN | CLOSED
  warning: string | null
```

## Output Requirements

### Required Output (DOC-003)

```yaml

  # === Identity ===
  issue_number: number          # GitHub Issue 编号
  repository: string            # owner/repo 格式
  dispatch_id: string           # 派发 ID
```

## Anti-Examples

### 反例 1: 过早关闭 Issue

**错误行为**:
```javascript
// ❌ WRONG: docs skill 关闭 Issue
async function syncIssueStatus(context) {
  await githubClient.postComment(owner, repo, issueNumber, comment);
  
  // ❌ 这是错误的！Issue 应该由管理层关闭
  if (result.status === 'SUCCESS') {
    await githubClient.updateIssue(owner, repo, issueNumber, { state: 'closed' });
  }
}
```

**为什么错误**:
1. **违反 BR-003**: 禁止过早关闭
2. **跳过验收**: 管理层没有机会审查
3. **无法返工**: Issue 已关闭，返工流程受阻

**正确做法**:
```javascript
// ✅ CORRECT: 只发布评论，不关闭 Issue
async function syncIssueStatus(context) {
  await githubClient.postComment(owner, repo, issueNumber, comment);
  
  // ✅ 不关闭 Issue，等待管理层验收
  // Issue 状态保持 OPEN
  
  return {
    comment_posted: true,
    issue_state_after: 'OPEN'  // 必须保持 OPEN
  };
}
```

---

### 反例 2: 不基于 Evidence 发布评论

**错误行为**:
```yaml
# ❌ WRONG: 评论内容没有 artifact 支撑
issue_progress_report:
  execution_status: SUCCESS
  # 没有引用 acceptance-decision-record
  # 没有引用 verification-report
  recommendation: PENDING_ACCEPTANCE
  # 无法追溯证据来源
```

**为什么错误**:
1. **违反 BR-001**: Evidence Consumption
2. **无法追溯**: 没有证据链
3. **可能造假**: 没有上游 artifact 验证

**正确做法**:
```yaml
# ✅ CORRECT: 基于 artifacts 发布评论
issue_progress_report:
  execution_status: SUCCESS
  artifacts_consumed:
    - contract_id: "RC-003"
      source: "specs/028/acceptance-decision-record.md"
      key_fields:
        decision: "APPROVED"
        must_fix_issues: []
    - contract_id: "TC-001"
      source: "specs/028/verification-report.md"
      key_fields:
        tests_passed: true
  recommendation: PENDING_ACCEPTANCE
  recommendation_reason: "Based on RC-003 (APPROVED) and TC-001 (tests pass)"
```

## Checklists

### Pre-Conditions
- [ ] dispatch context 包含 `issue_number`
- [ ] dispatch context 包含 `repository`
- [ ] GitHub API 可访问
- [ ] 收到上游角色的 artifacts

### Process Checks
- [ ] 读取了 acceptance-decision-record (RC-003)
- [ ] 读取了 execution result
- [ ] 生成了 DOC-003 compliant report
- [ ] 评论已发布到 GitHub
- [ ] **Issue 状态保持 OPEN** (BR-003)

### Post-Conditions
- [ ] Issue 评论已发布
- [ ] Issue 状态为 OPEN
- [ ] comment_url 已记录
- [ ] execution summary 已输出

### Forbidden Actions
- [ ] **未**调用 `updateIssue({ state: 'closed' })`
- [ ] **未**调用 `closeIssue()`
- [ ] **未**修改 Issue 标签
- [ ] **未**修改 Issue 标题

## Common Failure Modes

| 失败模式 | 表现 | 处理建议 |
|----------|------|----------|
| Issue 过早关闭 | docs skill 调用 closeIssue | 移除关闭逻辑，遵循 BR-003 |
| 无证据评论 | 评论内容无 artifact 引用 | 添加 artifact-reading 步骤 |
| 缺少 Issue 上下文 | dispatch 无 issue_number | 跳过 skill，记录 warning |
| API 调用失败 | GitHub API 返回错误 | 重试或记录 escalation |

## Notes

### 与 readme-sync 的关系
- `readme-sync`: 同步 README 到仓库
- `issue-status-sync`: 同步状态到 GitHub Issue
- 两者都是 docs 角色的同步职责

### 与 GitHub Issue Adapter 的关系
- Adapter 处理自动化模式（webhook 触发）
- issue-status-sync 处理手动模式（agent 执行后）
- 共用 `github-client.js` 和 `comment-templates.js`

### 与 acceptance 层的关系
- issue-status-sync **不替代** acceptance 决策
- 只报告执行结果，等待验收
- Issue 关闭由 acceptance 层决定

### 管理层验收流程
1. docs 发布 Issue 评论
2. Issue 保持 OPEN
3. 管理层审查执行结果 + artifacts
4. 管理层决策：
   - ACCEPT → 管理层关闭 Issue
   - REJECT → Issue 保持 OPEN，触发返工
