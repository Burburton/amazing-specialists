# Example: Stress Test

## 场景

对系统进行压力测试，找到性能极限和崩溃点。

## 测试配置

```yaml
stress_test:
  name: find_system_limit
  target: /api/*
  
  # 逐步加压
  stages:
    - { duration: '5m', target: 500 }   # 正常负载
    - { duration: '5m', target: 1000 }  # 2倍
    - { duration: '5m', target: 2000 }  # 4倍
    - { duration: '5m', target: 4000 }  # 8倍
    - { duration: '5m', target: 8000 }  # 16倍
    - { duration: '5m', target: 0 }     # 恢复
    
  # 关注指标
  focus:
    - 响应时间拐点
    - 错误率开始上升的点
    - 资源饱和点
```

## 测试结果分析

### 吞吐量 vs 并发

```
并发用户 | 吞吐量(RPS) | P99响应时间 | 错误率
---------|------------|-------------|--------
   500   |    2,500   |    180ms    |  0.01%
  1000   |    4,800   |    220ms    |  0.05%
  2000   |    8,500   |    350ms    |  0.12%
  4000   |   12,000   |    680ms    |  1.50%  ← 开始退化
  8000   |   10,500   |   2500ms    |  8.20%  ← 性能下降
```

### 关键发现

1. **最佳并发点**: 2000 用户，吞吐量最大且响应时间可接受
2. **性能拐点**: 4000 用户开始出现明显退化
3. **系统极限**: 约 4000-5000 并发用户
4. **崩溃特征**: 8000 用户时吞吐量反而下降（资源竞争）

### 资源使用

```yaml
at_4000_users:
  cpu: 92%        # 接近饱和
  memory: 85%     # 接近上限
  db_connections: 95/100  # 连接池几乎满
  
at_8000_users:
  cpu: 99%        # 完全饱和
  memory: 95%     # 接近 OOM
  db_connections: 100/100  # 连接池耗尽
  gc_pause: 800ms # GC 暂停过长
```

## 结论

1. 系统最大承载: 2000 并发用户（安全）
2. 性能拐点: 4000 并发用户
3. 建议配置: 服务器数量 x 2 或升级配置

## 优化建议

1. 增加数据库连接池
2. 优化内存使用，减少 GC 压力
3. 考虑水平扩展
