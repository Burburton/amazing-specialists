# 验收与返工机制设计

## 1. 文档目的

本文档定义“全自动产品研发闭环 MVP”中的 **验收（Acceptance）与返工（Rework）机制**。  
它解决的问题包括：

- 什么叫“完成”
- 什么叫“通过验收”
- 什么叫“失败但可返工”
- 什么情况下需要重规划
- 什么情况下必须升级到用户
- 验收包应该包含哪些内容
- 返工应该如何最小化、可控化、有限化
- 验收与返工如何和状态机、对象模型、管理层调度、OpenCode 专家包对接

本文件与以下文档配套：

- 《系统状态机与对象模型设计》
- 《OpenClaw 管理层调度设计》
- 《OpenCode 专家包设计》

前几份文档分别定义了：

- 系统怎么流转
- 管理层怎么调度
- 执行层怎么干活

本文档定义的是：

> **结果怎样被判断、失败怎样被修复、什么时候停、什么时候交给人来拍板。**

---

## 2. 设计目标

验收与返工机制必须满足以下目标：

1. **清晰**
   - 通过、失败、返工、重规划、升级的边界明确

2. **可执行**
   - 能直接映射成系统规则、状态迁移和对象字段

3. **最小返工**
   - 优先修复最小问题，不轻易推翻整轮产出

4. **可收敛**
   - 自动返工必须有次数上限和升级阈值，不能无限循环

5. **可解释**
   - 每次通过或失败都必须有明确理由和工件支撑

6. **用户友好**
   - 用户只在关键节点参与，不需要盯每个 task

7. **与角色化执行兼容**
   - 不同角色输出的结果都能进入统一验收与返工闭环

---

## 3. 核心概念定义

## 3.1 完成（Done）

“完成”表示某个对象已经完成了其定义范围内的工作，但不一定表示通过了更高层验证。

例如：

- developer task 完成：代码已实现并输出实现总结
- tester task 完成：测试已执行并输出结果
- reviewer task 完成：审查已完成并输出报告

**完成 ≠ 通过验收**

---

## 3.2 验证通过（Verified）

“验证通过”表示该对象已经满足了预定义的验证规则。

例如：

- build 通过
- 单测通过
- review 无 must-fix 项
- acceptance criteria 达到阈值

**验证通过是进入验收的前提，但不一定等于用户验收通过。**

---

## 3.3 验收通过（Accepted）

“验收通过”表示该阶段交付物满足当前 milestone 或阶段目标，并且在需要用户确认时已得到用户批准。

---

## 3.4 返工（Rework）

返工表示：

- 当前目标不变
- 当前范围大体不变
- 当前 milestone 仍合理
- 只需要修复局部问题或补齐缺口

---

## 3.5 重规划（Replan）

重规划表示：

- 原拆分方式、原实现路径、原 milestone 边界已不再合适
- 问题不是靠局部修补就能解决
- 需要回到 planning 层重新拆

---

## 3.6 升级（Escalation）

升级表示系统不能再自动决策，必须交给上层或用户判断。

升级分两种：

- **升级到管理层内部重协调**
- **升级到用户做决策**

---

## 4. 验收层在系统中的位置

整体闭环如下：

```text
Requirement
-> Spec
-> Milestone Planning
-> Task Dispatch
-> OpenCode Execution
-> Verification
-> Acceptance
-> Rework / Replan / Approve / Complete
```

验收层位于执行与下一轮推进之间，是整个系统的**质量闸门**。

它主要负责：

- 接收验证结果
- 形成阶段性判断
- 组织验收包
- 决定下一步是通过、返工、重规划还是升级
- 触发对应状态迁移

---

## 5. 设计原则

## 5.1 验收看“结果”，不看“努力程度”

实现很复杂、做了很多轮尝试，不等于可以放行。  
验收必须看：

- 目标是否满足
- 风险是否可接受
- 缺口是否可接受
- 当前阶段是否值得继续

## 5.2 先验证，再验收

没有通过最低验证门槛的结果，不应直接进入用户验收。

## 5.3 默认局部返工，避免整体推翻

能通过局部修复解决的问题，不要轻易进入重规划。

## 5.4 自动返工必须有限制

自动返工必须有：
- 最大次数
- 最大深度
- 最大连续失败阈值
- 风险升级阈值

## 5.5 用户只看阶段成果，不看底层噪音

用户应看到 milestone 级结果包，而不是所有 task 的细碎日志。

验收层主要围绕以下对象：

- `VerificationResult`
- `AcceptanceReport`
- `Escalation`
- `ReworkRequest`
- `ReplanRequest`

## 7. 验证层与验收层的关系

## 7.1 验证层负责“检查”

例如：
- build 是否通过
- test 是否通过
- review 是否通过
- security gate 是否通过

## 7.2 验收层负责“决策”

例如：
- 当前 milestone 是否可以交给用户看
- 是否只需要返工
- 是否必须重规划
- 是否可以继续下一阶段

### 一句话概括
- **Verification = 判断事实**
- **Acceptance = 做流程决策**

---

## 8. 验收分层设计

建议将验收分为三层：

### L1：Task-Level Acceptance
对单个 task 结果做局部可用性判断。

### L2：Milestone-Level Acceptance
对一组 task 构成的阶段性交付做综合判断。

### L3：Project-Level Acceptance
对整个项目或版本候选做最终性判断。

MVP 阶段主要实现 L1 + L2。  
L3 可先简化为“所有 milestone 完成后 project completed”。

---

## 9. Task 级验收设计

Task 级验收是最小闭环判断。

## 9.1 目标

判断一个 task 是否：
- 已完成
- 可进入下游
- 需要返工
- 需要升级

## 9.2 输入

- task object
- execution result
- generated artifacts
- related verification result
- task verification steps

## 9.3 输出

- `task_acceptance_status`
- 后续动作建议
- 是否进入下游角色
- 是否触发返工

## 9.4 Task 验收状态建议

```text
TASK_ACCEPTED
TASK_ACCEPTED_WITH_WARNINGS
TASK_REWORK_REQUIRED
TASK_REPLAN_REQUIRED
TASK_ESCALATED
```

## 9.5 判定规则

### TASK_ACCEPTED
满足：
- expected_outputs 已满足
- 关键 verification steps 全通过
- 无 must-fix 问题
- 风险在可接受范围内

### TASK_ACCEPTED_WITH_WARNINGS
满足：
- 主目标已达成
- 存在非阻塞性问题
- 不影响当前 milestone 推进
- 后续可通过 docs / cleanup / backlog 处理

### TASK_REWORK_REQUIRED
满足：
- 当前任务方向正确
- 问题可局部修复
- 不需要改 task 边界

### TASK_REPLAN_REQUIRED
满足：
- 当前 task 本身定义有问题
- 依赖结构不对
- 输出目标与上游设计冲突

### TASK_ESCALATED
满足：
- 风险高
- 多轮失败
- 依赖冲突无法自动化解决
- 缺少关键决策

---

## 10. Milestone 级验收设计

Milestone 级验收是用户感知的核心阶段。

## 10.1 验收目标

回答以下问题：

- 当前 milestone 的目标是否达成
- 哪些交付物已经具备可展示 / 可使用价值
- 剩余问题是否阻断继续推进
- 当前是否需要用户确认
- 下一步是继续、返工、重规划还是暂停

## 10.2 输入

- milestone object
- all related tasks
- task acceptance statuses
- verification results
- key artifacts
- review / security / docs summary
- acceptance criteria

## 10.3 输出

- `AcceptanceReport`
- milestone acceptance decision
- next action

## 10.4 Milestone 验收状态建议

```text
MILESTONE_ACCEPTED
MILESTONE_ACCEPTED_WITH_WARNINGS
MILESTONE_REWORK_REQUIRED
MILESTONE_REPLANNING_REQUIRED
MILESTONE_ESCALATED
```

## 10.5 判定规则

### MILESTONE_ACCEPTED
满足：
- 里程碑主要目标达成
- 所有 required tasks 均已通过
- 关键验证均通过
- 风险可接受
- 用户验收需要时已通过

### MILESTONE_ACCEPTED_WITH_WARNINGS
满足：
- 主要目标达成
- 有次要未完成项或低风险遗留
- 不影响进入下一 milestone
- 已记录待办和补偿措施

### MILESTONE_REWORK_REQUIRED
满足：
- 当前里程碑目标基本正确
- 缺口可通过补任务或局部返工弥补
- 无需重划 milestone 结构

### MILESTONE_REPLANNING_REQUIRED
满足：
- 当前里程碑范围不合理
- required tasks 定义错误
- 依赖关系或验收目标已不再适用

### MILESTONE_ESCALATED
满足：
- 需要用户拍板
- 自动系统无法合理平衡 trade-off
- 风险超过设定阈值

---

## 11. 验收门槛设计

## 11.1 最低通过门槛

MVP 建议设置统一基础门槛：

- required outputs 存在
- build 通过（如适用）
- 必需测试通过（如适用）
- review 无 must-fix
- 高风险任务完成所需专项检查
- acceptance criteria 覆盖度达到最低阈值

## 11.2 按任务类型调整门槛

### architecture task
必须：
- design note 存在
- 接口 / 边界定义清楚
- 主要风险列出
- implementation 可执行

### implementation task
必须：
- 代码改动存在
- 自检结果存在
- build / unit test / review 满足要求

### test task
必须：
- test report 存在
- 覆盖范围说明清楚
- pass / fail 结论明确

### review task
必须：
- review report 存在
- 结论明确
- must-fix / optional issue 区分明确

### docs task
必须：
- 文档更新结果存在
- 变更影响范围说明清楚

### security task
必须：
- security report 存在
- severity 级别清楚
- 是否可放行有明确结论

## 12. VerificationResult 详细设计

## 12.1 status 设计

建议统一为：

```text
PASS
PASS_WITH_WARNINGS
FAIL_RETRYABLE
FAIL_REPLAN
FAIL_ESCALATE
```

## 12.2 含义说明

### PASS
验证项全部满足。

### PASS_WITH_WARNINGS
验证项核心通过，但存在低风险问题。

### FAIL_RETRYABLE
当前失败可通过局部返工解决。

### FAIL_REPLAN
当前失败是结构性问题，建议回到规划层。

### FAIL_ESCALATE
当前失败已超出自动系统可裁定范围，需升级。

## 14. ReworkRequest 设计

返工不应该只是“打回去重做”，而应生成明确返工请求对象。

## 14.1 结构建议

```yaml

milestone_id:

```

## 14.2 关键字段说明

### scope
返工范围：
- task_only
- task_and_downstream
- milestone_partial

### reason_type
建议枚举：
- TEST_FAILURE
- REVIEW_REJECTION
- ACCEPTANCE_GAP
- DOCS_MISSING
- SECURITY_ISSUE
- PERFORMANCE_REGRESSION
- INCOMPLETE_OUTPUT

### required_fixes
必须修复项。

### non_goals
返工时不允许扩大的范围。

## 16. 失败分类体系

要让返工有效，必须先把失败分类做细。

## 16.1 一级分类

### A. 产出缺失类
例如：
- 缺少必须工件
- 输出格式不合规
- 未生成 expected outputs

### B. 质量不达标类
例如：
- 测试不通过
- review 不通过
- 文档不同步
- 实现不符合 spec

### C. 风险超标类
例如：
- 安全问题
- 权限模型错误
- 高风险依赖引入
- 性能回归严重

### D. 结构性问题类
例如：
- task 拆分不合理
- 当前方案与架构冲突
- acceptance criteria 不可实现
- milestone 范围错误

### E. 决策缺失类
例如：
- 缺少业务优先级判断
- trade-off 无法自动确定
- 用户目标不明确

---

## 16.2 二级分类建议

| 一级分类 | 二级分类示例 |
|---|---|
| 产出缺失 | missing_artifact, incomplete_output, invalid_format |
| 质量不达标 | build_failed, test_failed, review_rejected, docs_missing |
| 风险超标 | security_issue, perf_regression, dependency_risk |
| 结构性问题 | invalid_scope, dependency_cycle, design_mismatch |
| 决策缺失 | ambiguous_goal, unresolved_tradeoff, missing_user_decision |

---

## 17. 返工决策矩阵

## 17.1 task 级决策矩阵

| 失败类型 | 默认动作 |
|---|---|
| missing_artifact | REWORK |
| invalid_format | REWORK |
| build_failed | REWORK |
| test_failed | REWORK |
| review_rejected | REWORK |
| docs_missing | REWORK |
| security_issue_low | REWORK |
| perf_regression_low | REWORK |
| dependency_cycle | REPLAN |
| invalid_scope | REPLAN |
| unresolved_tradeoff | ESCALATE |
| ambiguous_goal | ESCALATE |

## 17.2 milestone 级决策矩阵

| 情况 | 默认动作 |
|---|---|
| 主要功能完成，少量低风险缺口 | ACCEPT_WITH_WARNINGS |
| required tasks 少量失败，但局部可补 | REWORK |
| 多个 required tasks 结构性失败 | REPLAN |
| 高风险专项不通过 | ESCALATE |
| 用户目标改变 | REPLAN |
| 自动返工次数耗尽 | ESCALATE |

## 18. 返工策略设计

## 18.1 基本原则

1. 返工优先修最小闭环
2. 返工要保留失败上下文
3. 返工不能默认扩大范围
4. 返工必须带有清晰的 required fixes
5. 返工超过阈值必须升级

## 18.2 最小返工策略

返工优先级：

1. 修 artifact 缺失或格式问题
2. 修单点逻辑错误
3. 修边界条件和回归缺陷
4. 再考虑重排局部 task
5. 最后才进入重规划

## 18.3 返工输入要求

返工派发给专家角色时，必须至少包含：

- 原 task payload
- 原 execution result
- failed checks
- review comments / test logs
- required fixes
- non-goals
- retry count

---

## 19. 自动返工上限设计

## 19.1 task 级返工上限

建议：
- 普通 task：`max_retry = 2`
- 中风险 task：`max_retry = 1`
- 高风险 task：`max_retry = 0~1`，优先升级

## 19.2 milestone 级返工上限

建议：
- milestone 局部返工轮次不超过 2
- 超过则必须进入 replan 或 escalate

## 19.3 项目级连续失败阈值

建议：
- 连续 3 次 milestone 返工未收敛 -> 项目自动暂停并升级

---

## 20. 升级机制设计

## 20.1 升级到管理层

适用于：
- task role 选错
- 需要增加 reviewer / tester / security 追加检查
- 需要局部新增 task
- 需要局部重排依赖

## 20.2 升级到用户

必须升级给用户的典型情况：

- 目标冲突
- 范围变化
- 预算 / 成本 / 质量三者不可兼得
- 架构 trade-off 无法自动决策
- 高风险问题是否接受存在争议
- 连续失败后是否继续该 milestone

```yaml
impact_if_stop:

```

## 21. 用户验收机制详细设计

## 21.1 什么时候触发用户验收

MVP 建议仅在这些节点触发：

1. spec 初版完成
2. 每个 milestone 完成
3. 重大方案变更
4. 高风险问题需要接受或放弃
5. 自动返工达到上限
6. 版本候选完成

## 21.2 用户不应看到什么

用户不应被这些信息淹没：

- 所有中间日志
- 所有临时失败尝试
- 每个 task 的细粒度状态波动
- 内部 prompt 或推理细节

## 21.3 用户应看到什么

用户验收包应包含：

- 当前 milestone 目标
- 已完成内容
- 关键可展示成果
- 主要未完成项
- 测试与 review 总结
- 风险摘要
- 需要用户决策的点
- 系统推荐动作

## 23. Recommendation 设计

验收层最终需要给出系统建议。

## 23.1 recommendation 枚举

```text
CONTINUE
CONTINUE_WITH_WARNINGS
REWORK
REPLAN
PAUSE
STOP
```

## 23.2 使用规则

### CONTINUE
当前 milestone 通过，可以进入下一阶段。

### CONTINUE_WITH_WARNINGS
当前 milestone 可进入下一阶段，但有遗留项。

### REWORK
当前 milestone 不应继续推进，先局部返工。

### REPLAN
当前 milestone / task 结构需重新规划。

### PAUSE
当前不宜继续，等待额外信息或用户决策。

### STOP
当前项目或当前路径不值得继续。

## 24. 与 OpenCode 角色的衔接

不同角色的输出会进入不同验收路径。

## 24.1 architect 输出的验收重点
- 方案是否清晰
- 边界是否明确
- 风险是否列出
- implementation 是否可执行

## 24.2 developer 输出的验收重点
- 是否真的实现目标
- 是否超范围改动
- 自检是否充分
- 是否存在未披露问题

## 24.3 tester 输出的验收重点
- 关键路径是否覆盖
- pass/fail 结论是否明确
- gap 是否清楚
- failure 是否可复现

## 24.4 reviewer 输出的验收重点
- must-fix 是否关闭
- 风险是否可接受
- 建议是否可执行

## 24.5 docs 输出的验收重点
- 用户侧影响是否已同步
- 关键技术文档是否更新

## 24.6 security 输出的验收重点
- 是否存在 high / critical 问题
- 是否允许继续推进

## 25. 与状态机的衔接

## 25.1 task 级

```text
IMPLEMENTED / TESTED / REVIEWED
-> VERIFIED
-> DONE
```

失败路径：

```text
FAILED_TEST / CHANGES_REQUESTED / FAILED_VERIFICATION
-> REWORK_REQUESTED
-> IN_PROGRESS
```

## 25.2 milestone 级

```text
IN_PROGRESS
-> VERIFYING
-> READY_FOR_ACCEPTANCE
-> ACCEPTED
```

失败路径：

```text
VERIFYING -> REWORK_REQUIRED
VERIFYING -> REPLANNING_REQUIRED
READY_FOR_ACCEPTANCE -> REWORK_REQUIRED
READY_FOR_ACCEPTANCE -> REPLANNING_REQUIRED
```

## 25.3 project 级

```text
ACCEPTANCE_READY
-> WAITING_FOR_USER_ACCEPTANCE
-> APPROVED
-> READY_FOR_EXECUTION / COMPLETED
```

## 26. 与管理层调度的衔接

验收与返工机制需要驱动管理层做以下动作：

- 创建 ReworkRequest
- 创建 ReplanRequest
- 重新选择角色
- 增加附加检查 task
- 生成 AcceptanceReport
- 生成 Escalation 包
- 暂停或继续项目

### 典型交互

#### 例 1：developer 实现后 reviewer 拒绝
- reviewer 输出 `FAILED_RETRYABLE`
- 验收层生成 ReworkRequest
- Dispatch Coordinator 重新派发给 developer

#### 例 2：多个 task 连续失败
- 验收层判定 `REPLAN`
- Milestone Planner 重新拆 task

#### 例 3：安全问题严重
- 验收层输出 `ESCALATE`
- 用户决定是否接受风险或调整需求

## 27. 规则引擎建议

建议将验收与返工逻辑做成规则引擎，而不是散落在各处 if/else。

## 27.1 规则类别

### gate rules
决定能否通过。

### rework rules
决定是否返工。

### replan rules
决定是否重规划。

### escalation rules
决定是否升级。

### acceptance package rules
决定用户看到什么内容。

## 27.2 规则示例

```yaml
- rule_id: require_review_for_code_change
  when:
    task.type in [IMPLEMENTATION, BUGFIX, REFACTOR]
  then:
    review_required: true
```

```yaml
- rule_id: security_blocker
  when:
    security_report.contains_high_severity_issue == true
  then:
    acceptance_status: MILESTONE_ESCALATED
```
